内容简介 ······

学习编程的人都知道，阅读、剖析名家代码乃是提高水平的捷径。源码之前，了无秘密。大师们的缜密思维、经验结晶、技术思路、独到风格，都原原本本体现在源码之中。

这本书所呈现的源码，使读者看到vector的实现、list的实现、heap的实现、deque的实现、Red Black tree的实现、hash table的实现、set/map的实现；看到各种算法（排序、查找、排列组合、数据移动与复制技术）的实现；甚至还能够看到底层的memory pool和高阶抽象的traits机制的实现。

    其中T1是需要判断的参数的类型。
    之前虽然知道可以用“是否可以作为union的成员”的准则来判断一个类型是否为POD类型，但是总感觉这个方法用在这儿有些不妥：用这个方法的话，如果不是POD类型，则编译出错，而这里显然是“根据是否为POD类型调用不同的函数”，如果用“编译出错”的方法显然太暴力且不符合该场景。所以就很好奇地看了一下STL的源代码，发现其实方法很简单：
    在stl的type_traits.h文件中，定义了一个结构体模板 __type_traits：
    template <class _Tp>
    struct __type_traits {
    ................
    typedef __false_type is_POD_type;
    };
    之后又定义了一系列该模板的特化版本：
    __STL_TEMPLATE_NULL struct __type_traits<char> {
    ................
    typedef __true_type is_POD_type;
    };
    __STL_TEMPLATE_NULL struct __type_traits<signed char> {
    ................
    typedef __true_type is_POD_type;
    };
    .......................
    从代码中很容易看出来，一旦__type_traits的模板参数为POD类型的，编译器会找到相应的特化版本，其is_POD_type就是__true_type；而如果不是POD的，则会找到泛化版本，其is_POD_type就是__false_type。如此一来，就实现了在避免编译期报错的情况下，判断一个类型是否为POD了。真是十分巧妙！！
    看来还是得多读优秀的代码，以及加强对C++泛型的理解。。。侯捷没有在书中展开来说这个，估计是他觉得这个太简单了吧。。。。
