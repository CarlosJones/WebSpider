    简单总结如下：
    1. 系统默认为每个线程的栈保留1MB的地址空间，提交2个页面的物理内存。但这两个设置可以在链接的时候传入/STACK参数修改。
    2. 当前已提交页面的最后页面始终带有一个保护属性。随着函数调用层次的增加，当线程试图访问这个带保护属性的页面时（往往意味着当前已提交内存将用尽），系统将得到一个通知，去掉这最后一个页面保护属性，并将此页面的下一个页面提交物理内存，此时，这个新页面成为最后一个已提交页面，获得保护属性。
    需要注意的是，有时当前函数就所占用内存超过了已提交页面和保护页面之和的大小，这样线程有可能直接访问保护页面之后的保留页面，这样可能引发访问违规。
    为了解决这个问题，当检测到当前函数需要的内存大小超过了单个页面大小，编译器会自动插入一段堆栈检测函数，该函数根据函数所需的栈大小初始化栈内的页面。例如，当前函数需要16KB大小内存，最终检测代码会提交4个页面，并将保护属性置于第5个页面。
    3. 当线程访问到栈底部的前一个页面时，系统除了提交该页面外，还会抛出一个stack overflow的异常，表明所有可用空间都已被占用。程序此时可以通过SEH捕获该异常并作出处理。如果栈继续增长，当线程试图访问栈底部地址时，会引发一个访问违规异常，导致整个进程被系统强行终止。
    4. 另两种形式的溢出：
    下溢：
    int main(){
    int i[1] ;
    i[10000] = 0;
    } 线程试图访问超过栈顶地址的内存。
    上溢：
    int main(){
    int i[10000000];
    i[0] = 0;
    }
    另：x86架构的设计，堆栈是由高地址向低地址增长。ARM貌似支持两种增长方式。
