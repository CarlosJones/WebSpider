Let Over Lambda is one of the most hardcore computer programming books out there. Starting with the fundamentals, it describes the most advanced features of the most advanced language: Common Lisp. Only the top percentile of programmers use lisp and if you can understand this book you are in the top percentile of lisp programmers. If you are looking for a dry coding manual that re-hashes common-sense techniques in whatever langue du jour, this book is not for you. This book is about pushing the boundaries of what we know about programming. While this book teaches useful skills that can help solve your programming problems today and now, it has also been designed to be entertaining and inspiring. If you have ever wondered what lisp or even programming itself is really about, this is the book you have been looking for.

    U-Language
    这一节讲本书排版的约定，我从来没有这么认真的读过一本编程书中介绍排版约定的章节。因为内容远不止排版那么简单。
    这一节主要向 Haskell.咖喱 致敬。（抱歉 Curry 大神，我每次都忍不住）
    Haskell Curry 是 《Foundations Of Mathematical Logic》的作者，主要贡献是 Combinatory logic。著名纯函数式语言 Haskell 以其命名，curry 函数的概念也是从其名字里来的。
    Curry 不但尝试形式化ideas ，而且对于形式化的表达方式也尝试去形式化。因此他设定了一个假想的语言 U-Language ，作为作者与读者之间沟通的标准。
    在 Curry 的术语中，macro 被描述为 meta-programming 。一个 meta-prog 是用来帮助程序员更好的写程序的。其他的编程语言中也有meta-prog 的概念，比如最近比较火的那本 Ruby元编程 ，但是没有哪个语言像lisp那样完全的接受meta-prog。
    在LISP语言本身中编写 meta-language 这个决定，是LISP具有的生产力优势的关键。
    在书里经常需要拿其他语言对比时，为了减少树敌，一律称为 Blub 。即：不具备LISP Macro 的语言。这是从 Paul Graham 的《Beat the average》一文中引来的。（喂，这样树敌更多吧，简直是地图炮啊）
    编写复杂的macro 是一个迭代的过程。没有哪个lisp程序员会坐下来随手写一大篇lisp代码。因为：1. LISP每页代码中包含的信息量更大 2. LISP技术鼓励程序员让代码不断生长，为应用的需要对语言做一系列的改进。
    ========== Section 1.3 The Lisp Utility
    LISP程序就像是一个洋葱，上层程序构建在下层工具之上。同样，这本书也像是洋葱，它构建在 《On Lisp》之上。
    这一节介绍了几个后面写macro用得着的工具函数：
    mkstr 将参数输出成字符串
    symb 创建 symbol
    group 将列表分组
    flatten 将tree 展平
    fact factorial
    choose 二项式系数函数

    第一章 第一节，介绍本书的主题 Macros。这一节更像是一个LISP hacker写给那些向往LISP的人的信。
    深刻理解掌握 Macro 是从LISP中级水平步入LISP专家的关口。
    首先要理解为什么需要Macro ： 不经过小心设计，以及没有艰苦努力的大型程序，到处都充满“冗余的模式”和“不灵活的抽象”。 在写程序的过程中，识别这些模式及抽象，然后寻找一种方法来自动化它们，也即 code write code。
    Macro 让程序员可以在更高的抽象层次上写程序，所有的（大部分吧）语言都是一层皮。其他不具备macro功能的语言，是僵化、特定的，而Lisp是流动、通用的。
    Age of Macro is Coming （我怀疑）
    一些关于Macro的说法：非必要时不用；破坏纯FP风格。不理解才需要编程风格。
    类比： LISP -》Macro ， C -》指针。
    Paul Graham 的 《On LISP》必读。
    macro 的前沿研究领域： combinations of macros 组合子。
    macro 既不学术，也不商业。

    Macros are a hacker's best friend.
    敬畏LISP的威力。

    Chapter 2 Closures
    Closure = Let + Lambda = Program ！ （我的理解正确么？请指正我）
    Let 用来构造数据的binding，组织 data
    Lambda 用来构造操作数据的函数，组织 code
    程序就是对data 和 code 的组织。
    也即：程序=数据结构+算法。
    ==================================================
    2.1 Closure-Oriented Programming
    closure 中文翻译为“闭包”，这个翻译很糟糕，完全不知所谓，而且与其他的概念相冲突（见 SICP）。以前我也被闭包这个词迷惑了很久，不得要领。
    closure 有时被称为： saved lexical environment ， lisper 称： let over lambda
    在perl 和 javascript 中，也需要掌握 closure 的概念。
    closure 的概念很难懂，因为他们很简单。（围棋的规则也很简单）
    closure 与 object ： closure是基础的，而object 概念不是, 通过closure可以构造object和各种复杂的class的行为。
    相对object， closure 是组织 data 和 code 更简单直接的方法。
    关于CLOS，它有很多高级特性，但是很少需要用到。
    ==================================================
    2.2 Environments and Extent
    indefinite extent: 怎么翻译？ 非限定生命期？有点类似Java 中 new 一个对象，没有ref后可以被gc掉。
    我么创建一个环境（environment）后，可以在后面任何时刻继续访问。即在heap上分配，而非在stack 上分配。LISP默认使用此种 indefinite extent。效率虽然有些许损失，但好处大大的，而且还可以通过编译器优化，或人为通过 declare指定优化。
    在LISP中，变量没有类型，只有值具有类型。
    要创建一个 indefinite extent 的值，最简单的办法是用 cons cell。
    ==================================================
    2.3 Lexical and Dynamic Scope
    用于表示变量在何处可以访问的术语是作用域（scope），在现代语言中词法作用域最常用。
    词法作用域，即只在代码段范围内可用。
    let 创建bindings，引入词法作用域变量。其他如 lambda ， defun 的参数也是 lexical variable。
    组合使用 indefinite extent 和 lexical scoping 。 lexical scope 与 Closure 非常相关，因此经常称为： lexical closures。
    Dynamic Scope : LISP中的黑话， temporary extent + global Scope
    special variables: 可以unbound ；可以被re-bound, shadow ； 运行时确定 ; 持续关联到symbol
    lexical variables: 不可unbound，总有值；编译时确定; symbol只在编译时用，编译完就没symbol的事了。
    (defvar temp-special)
    (setq temp-special 1)
    (defun temp-special-returner () temp-special)
    (let ((temp-special 2)) ;; shadow
    (temp-special-returner))
    > 2
    TEMP-SPECIAL
    > 1
    (symbol-value 'TEMP-SPECIAL)
    > 1
    在 let binding special variable 时，会先保存原值，用新值overwrite，用完后再恢复原值。
    Common Lisp转向了 lexical scope, 替换掉 Dynamic scope 是件好事。special 毕竟是特殊的，尽量不用吧。
    ==================================================
    2.4 Let It Be Lambda
    let 创建环境，binding names，返回最后一个expr 的值。let需要一个数据结构来保存指向值的指针。
    lambda 并不是函数, function函数可以把 lambda 表达式转成函数，common lisp 中有一个 lambda macro：
    (function '(lambda (x) x)) => #'(lambda (x) x) => (lambda (x) x)
    (let ((y 1))
    (lambda (x)
    (incf y x)))
    调用时，创建environment，其中包含指向 lambda function，返回 closure。
    ==================================================
    2.5 Let Over Lambda
    经过前面的铺垫，终于引出了本书的核心概念了。
    (let
    (lambda ...))
    let over lambda = lexical Closure
    let创建 lexical environment, lambda 函数操作它。
    (let [state]
    (lambda ..)) ==> [ state <- code ]
    function with state, objects.
    let over lambda 可以实现object 的数据+操作封装。
    ==================================================
    2.6 Lambda Over Let Over Lambda
    (lambda
    (let
    (lambda ...)))
    ==>
    (defun counter-class ()
    (let ((counter 0))
    (lambda () (incf counter))))
    Closures are the building blocks of lisp.
    CLOS: do not use it when a lambda will suffice.
    Closure 实际使用例子，block-scanner ：判断一个string是否出现，因为有可能跨两个甚至多个block，因此函数需要保存状态。
    ==================================================
    2.7 Let over Lambda over Let over Lambda
    (let 【share-states】 ;; class variable
    (lambda
    (let [state]
    (lambda ...))))
    在多个 Closure 之间共享状态。
    let 、lambda 是基础概念，而object、class 是衍生的概念。
    Object-Oriented 是一种固定的组织data，code的方式，而 let, lambda 可以更灵活的组织。

    Lambda 用来构造操作数据的函数，组织 code
    程序就是对data 和 code 的组织。
    也即：程序=数据结构+算法。
    ==================================================
    2.1 Closure-Oriented Programming
    closure 中文翻译为“闭包”，这个翻译很糟糕，完全不知所谓，而且与其他的概念相冲突（见 SICP）。以前我也被闭包这个词迷惑了很久，不得要领。
    closure 有时被称为： saved lexical environment ， lisper 称： let over lambda
    在perl 和 javascript 中，也需要掌握 closure 的概念。
    closure 的概念很难懂，因为他们很简单。（围棋的规则也很简单）
    closure 与 object ： closure是基础的，而object 概念不是, 通过closure可以构造object和各种复杂的class的行为。
    相对object， closure 是组织 data 和 code 更简单直接的方法。
    关于CLOS，它有很多高级特性，但是很少需要用到。
    ==================================================
    2.2 Environments and Extent
    indefinite extent: 怎么翻译？ 非限定生命期？有点类似Java 中 new 一个对象，没有ref后可以被gc掉。
    我么创建一个环境（environment）后，可以在后面任何时刻继续访问。即在heap上分配，而非在stack 上分配。LISP默认使用此种 indefinite extent。效率虽然有些许损失，但好处大大的，而且还可以通过编译器优化，或人为通过 declare指定优化。
    在LISP中，变量没有类型，只有值具有类型。
    要创建一个 indefinite extent 的值，最简单的办法是用 cons cell。
    ==================================================
    2.3 Lexical and Dynamic Scope
    用于表示变量在何处可以访问的术语是作用域（scope），在现代语言中词法作用域最常用。
    词法作用域，即只在代码段范围内可用。
    let 创建bindings，引入词法作用域变量。其他如 lambda ， defun 的参数也是 lexical variable。
    组合使用 indefinite extent 和 lexical scoping 。 lexical scope 与 Closure 非常相关，因此经常称为： lexical closures。
    Dynamic Scope : LISP中的黑话， temporary extent + global Scope
    special variables: 可以unbound ；可以被re-bound, shadow ； 运行时确定 ; 持续关联到symbol
    lexical variables: 不可unbound，总有值；编译时确定; symbol只在编译时用，编译完就没symbol的事了。
    (defvar temp-special)
    (setq temp-special 1)
    (defun temp-special-returner () temp-special)
    (let ((temp-special 2)) ;; shadow
    (temp-special-returner))
    > 2
    TEMP-SPECIAL
    > 1
    (symbol-value 'TEMP-SPECIAL)
    > 1
    在 let binding special variable 时，会先保存原值，用新值overwrite，用完后再恢复原值。
    Common Lisp转向了 lexical scope, 替换掉 Dynamic scope 是件好事。special 毕竟是特殊的，尽量不用吧。
    ==================================================
    2.4 Let It Be Lambda
    let 创建环境，binding names，返回最后一个expr 的值。let需要一个数据结构来保存指向值的指针。
    lambda 并不是函数, function函数可以把 lambda 表达式转成函数，common lisp 中有一个 lambda macro：
    (function '(lambda (x) x)) => #'(lambda (x) x) => (lambda (x) x)
    (let ((y 1))
    (lambda (x)
    (incf y x)))
    调用时，创建environment，其中包含指向 lambda function，返回 closure。
    ==================================================
    2.5 Let Over Lambda
    经过前面的铺垫，终于引出了本书的核心概念了。
    (let
    (lambda ...))
    let over lambda = lexical Closure
    let创建 lexical environment, lambda 函数操作它。
    (let [state]
    (lambda ..)) ==> [ state <- code ]
    function with state, objects.
    let over lambda 可以实现object 的数据+操作封装。
    ==================================================
    2.6 Lambda Over Let Over Lambda
    (lambda
    (let
    (lambda ...)))
    ==>
    (defun counter-class ()
    (let ((counter 0))
    (lambda () (incf counter))))
    Closures are the building blocks of lisp.
    CLOS: do not use it when a lambda will suffice.
    Closure 实际使用例子，block-scanner ：判断一个string是否出现，因为有可能跨两个甚至多个block，因此函数需要保存状态。
    ==================================================
    2.7 Let over Lambda over Let over Lambda
    (let 【share-states】 ;; class variable
    (lambda
    (let [state]
    (lambda ...))))
    在多个 Closure 之间共享状态。
    let 、lambda 是基础概念，而object、class 是衍生的概念。
    Object-Oriented 是一种固定的组织data，code的方式，而 let, lambda 可以更灵活的组织。
    推荐

    U-Language
    这一节讲本书排版的约定，我从来没有这么认真的读过一本编程书中介绍排版约定的章节。因为内容远不止排版那么简单。
    这一节主要向 Haskell.咖喱 致敬。（抱歉 Curry 大神，我每次都忍不住）
    Haskell Curry 是 《Foundations Of Mathematical Logic》的作者，主要贡献是 Combinatory logic。著名纯函数式语言 Haskell 以其命名，curry 函数的概念也是从其名字里来的。
    Curry 不但尝试形式化ideas ，而且对于形式化的表达方式也尝试去形式化。因此他设定了一个假想的语言 U-Language ，作为作者与读者之间沟通的标准。
    在 Curry 的术语中，macro 被描述为 meta-programming 。一个 meta-prog 是用来帮助程序员更好的写程序的。其他的编程语言中也有meta-prog 的概念，比如最近比较火的那本 Ruby元编程 ，但是没有哪个语言像lisp那样完全的接受meta-prog。
    在LISP语言本身中编写 meta-language 这个决定，是LISP具有的生产力优势的关键。
    在书里经常需要拿其他语言对比时，为了减少树敌，一律称为 Blub 。即：不具备LISP Macro 的语言。这是从 Paul Graham 的《Beat the average》一文中引来的。（喂，这样树敌更多吧，简直是地图炮啊）
    编写复杂的macro 是一个迭代的过程。没有哪个lisp程序员会坐下来随手写一大篇lisp代码。因为：1. LISP每页代码中包含的信息量更大 2. LISP技术鼓励程序员让代码不断生长，为应用的需要对语言做一系列的改进。

    第一章 第一节，介绍本书的主题 Macros。这一节更像是一个LISP hacker写给那些向往LISP的人的信。
    深刻理解掌握 Macro 是从LISP中级水平步入LISP专家的关口。
    首先要理解为什么需要Macro ： 不经过小心设计，以及没有艰苦努力的大型程序，到处都充满“冗余的模式”和“不灵活的抽象”。 在写程序的过程中，识别这些模式及抽象，然后寻找一种方法来自动化它们，也即 code write code。
    Macro 让程序员可以在更高的抽象层次上写程序，所有的（大部分吧）语言都是一层皮。其他不具备macro功能的语言，是僵化、特定的，而Lisp是流动、通用的。
    Age of Macro is Coming （我怀疑）
    一些关于Macro的说法：非必要时不用；破坏纯FP风格。不理解才需要编程风格。
    类比： LISP -》Macro ， C -》指针。
    Paul Graham 的 《On LISP》必读。
    macro 的前沿研究领域： combinations of macros 组合子。
    macro 既不学术，也不商业。

    Macros are a hacker's best friend.
