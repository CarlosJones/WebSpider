《Android开发精要》如何才能写出贴近Android设计理念、能够更加高效和可靠运行的Android应用？通过Android的源代码去了解其底层实现细节是最重要的方法之一！然而，Android系统太过于庞大，源代码实现复杂，学习的技术门槛和时间成本都很高。有没有一种方法既能帮助开发者深入理解Android应用开发，又能不被底层大量的实现细节所羁绊，迅速掌握编写高质量Android应用所需的知识？《Android开发精要》针对这个问题给出了完美的答案！它从Android繁杂的源代码中抽取出了Android开发的“精华”和“要点”，剥离了大量琐碎的底层实现细节，进行了高度概括和总结，不仅能帮助开发者迅速从宏观上理解整个Android系统的设计理念，而且能帮助开发者迅速从微观上掌握核心知识点的原理，从而编写出高质量的Android应用。

《Android开发精要》共13章，在逻辑上分为4个部分。第一部分（1-2章）：第1章系统介绍了Android的系统架构、核心模块和设计思想，旨在让读者真正理解它的设计理念；第2章讲解了Android源代码的获取、编译、阅读和编辑的方法。第二部分（3-6章）：第3章深入阐述了Android组件机制的设计理念和重要特征，并详细介绍了4大组件的方法和原理；第4章讲解了Intent对象和Android的意图机制，阐明了Android是如何将来自不同应用、不同进程的组件整合在一起的；第5章解析了Android中各个组件的生命周期，包括组件的进程模型和线程模型；第6章从开发的角度详细阐述了组件间数据传递的解决方案，以及它们的优缺点和适用场景。第三部分（7-8章）：第7章深入讲解了Android的控件框架，结合实际项目对重要控件的实现和使用逐一进行了分析，还包含Android 4.0界面开发的实践“精华”；第8章分析了Android的资源体系，剖析了Android底层对资源的处理。第4部分（9-13章）：第9章讲解了Android的数据存储结构，以及不同数据存储模式的使用要点；第10章分析了Android的各种网络连接方式，涵盖NFC和基于Wifi的P2P连接等内容；第11章讲解了Android的定位服务、地址解析服务、地图服务的框架实现；第12章仔细分析了Android各种多媒体功能的实现机制；第13章对Android中比较有特色的一些模块的实现细节进行了分析。

    一、主要负责：与用户进行交互的组件
    二、设计理念，更近于Web页面
    （1）运行模式：多tab页-多任务；页面栈的当前页面-任务的界面组件栈 栈顶组件
    （2）定位&切换管理
    定位：url-各种描述信息（类型信息|数据uri|数据类型信息）；
    切换：页面间的切换和数据传输 通过服务器 vs.
    界面组件的切换 依赖Android组件管理服务；
    （3）界面组件的功能设计
    都近于黑盒设计；
    cookie来存储 vs. 应用环境对象（Application Context），小数据量的共享数据
    三、界面组件的开发
    继承自 android.app.Activity 或其子类
    与界面组件相关的问题：
    （1）构造界面：第一要务。
    界面样式和内容：R类（资源的目录列表）；Activity.setContentView()展示样式
    （2）处理交互事件：
    全局事件
    具体控件的事件：观察者模式，监听 eg. View.setOnClickListener()
    （3）构造交互资源，eg.菜单、对话框等. 会耗时，阻塞主进程。so 要注意效率。
    （4）管理数据：多任务，进程托管，so 组件的生命周期管理,很重要。
    （5）配置任务模型：同一任务中的界面组件-依照栈模型线性排列。
    so 要合理配置界面组件的任务模型
    （6）适应配置环境变化：
    eg.硬键盘消隐、屏幕朝向、语言环境；
    处理：会销毁重建-界面组件，so双刃剑；
    配置：通过activity的配置项configChanges来配置，并在Activity.onConfigurationChanged()中详细处理
    四、界面组件的数据结构
    （1）android.app.Activity的继承结构，P40
    （2）派生自android.content.Context类，提供了运行的基本环境，是通信的桥梁（各组件与系统服务）。
    （3）Context类隐藏了，应用与系统服务通信的细节，简化了上层应用的开发。

    架构
    应用部分：包含在Anroid设备上运行的所有应用，他们是Android系统中直接面向用户的部分
    核心部分：Android系统中核心的功能实现，包括应用框架 ，核心类库等，每个Android应者的开发者，都在此基础上进行应用开发的。
    底层部分：Andorid系统中核心的功能实现，包括应用框架，核心类库等，每个Andorid应用的开发者，都是在此基础上进行应用开发的。
    应用层：各种app
    核心层：框架层（组件管理服务，窗口管理服务，系统数据源组件，控件框架，资源管理服务，安装包管理服务），核心类库（libe，SQLite，Webkit,OpenGL,FreeType） ，运行时(Java核心类库，虚拟机)
    底层部分：硬件抽象层（音视频接口，GPS接口，通话接口，Wifi接口）
    Linux内核：linux内核（Linux Kernel）（进程/线程 ，电源管理，驱动）
    应用层：
    权限：

    开发大规模运算图形处理等要用的开源C/C++类库的。用JNI接口与上层java实现交互，利用Android提供的交叉编译工具生成类库并添加到应用。
    用户可以直接用Andorid系统的特定类库，Andorid提供了NDK，它由C/C++的一些接口构成，开发者可以通过它更搞笑的调用特定的系统功能。

    关于通信：组件组成应用，组件通信通过框架曾的系统服务，集中的调度和传递信息。
    运行时：
    运行时组成：java核心类库+java虚拟机Dalvik。
    核心库：从框架曾到应用层要用到的基础java库，java对象库，文件管理库，网络通信库等。
    Dalvik是为Andorid量身打造的java虚拟机，动态解析应用，分配空间，管理对象生命周期等。
    Dalvik是采取寄存器的虚拟框架设计。

    核心类库
    SDk的变迁，由二进制动态库共同构成，通常使用C/C++进行开发。
    运行原理：被系统服务加载到进程空间，通过类库提供的JNI接口进行调用。
    核心类库的来源：原生类库+第三方类库（JNI）
    第三方类库 要通过核心库提供的NDK形式提供给开发者
    硬件抽象层和Linunx内核
    搭建在Linux内核的。

    控件属性：焦点，可视性，可用性，标石，背景灯属性。
    事件：为控件添加交互事件监听对象（Event Listener）观察模式
    完整的应用资源：所有和界面有关的元素，比如界面布局，文字信息，尺寸信息，颜色和图像都从代码中剥离出来，用应用资源来进行描述。（资源目录，xml资源文件，数据资源文件构成）
    数据存储：
    本地存储+云存储
    本地存储不能被访问

    全局事件通知机制。消息推送功能，android.app.NotificationManager对象将消息送到系统的状态栏。

    在Activity.onDestroy函數中回收線程仍不夠完美，因為在資源緊張的情況下，系統會強制回收組件，此時該組件的Activity.onDestroy函數可能會沒有調用，從而導致線程洩漏。
    一個更好的線程管理方案是將線程的句柄信息當作界面組件的狀態信息緩存下來。如果系統強行回收該組件對象，則需要在組件再次構造時，根據緩存線程的句柄信息找到該線程，從而避免線程洩露。
    /* 線程管理基本框架 */
private static final String WORKER_KEY = "thread_id";
private Thread worker; // 一個後臺線程，執行耗時操作

protected void onCreate(Bundle savedInstanceBundle) {
    // 如果組件不是重新構造，嘗試找到先前的線程
    if (savedInstanceBundle != null) {
        final long threadId = savedInstanceBundle.getLong(WORKER_KEY);
        worker = findThreadById(threadId);
    }

    // 如果沒有老的線程，構造新的
    if (worker == null) {
        worker = createNewWorker();
        worker.start();
    }
}

protected void onSaveInstanceState(Bundle outState) {
    // 如果後臺線程還在運行，緩存其線程id
    if (worker != null) {
        outState.putLong(WORKER_KEY, worker.getId());
    }
}

protected void onDestroy() {
    // 組件生命週期終止，停止線程執行，釋放線程資源
    if (worker != null) {
        worker.interrupt();
        worker == null;
}

    在Activity.onDestroy函數中回收線程仍不夠完美，因為在資源緊張的情況下，系統會強制回收組件，此時該組件的Activity.onDestroy函數可能會沒有調用，從而導致線程洩漏。
    一個更好的線程管理方案是將線程的句柄信息當作界面組件的狀態信息緩存下來。如果系統強行回收該組件對象，則需要在組件再次構造時，根據緩存線程的句柄信息找到該線程，從而避免線程洩露。
    /* 線程管理基本框架 */
private static final String WORKER_KEY = "thread_id";
private Thread worker; // 一個後臺線程，執行耗時操作

protected void onCreate(Bundle savedInstanceBundle) {
    // 如果組件不是重新構造，嘗試找到先前的線程
    if (savedInstanceBundle != null) {
        final long threadId = savedInstanceBundle.getLong(WORKER_KEY);
        worker = findThreadById(threadId);
    }

    // 如果沒有老的線程，構造新的
    if (worker == null) {
        worker = createNewWorker();
        worker.start();
    }
}

protected void onSaveInstanceState(Bundle outState) {
    // 如果後臺線程還在運行，緩存其線程id
    if (worker != null) {
        outState.putLong(WORKER_KEY, worker.getId());
    }
}

protected void onDestroy() {
    // 組件生命週期終止，停止線程執行，釋放線程資源
    if (worker != null) {
        worker.interrupt();
        worker == null;
}

    界面組件acitvity和web頁面相比，在設計理念上，有如下相似之處：
    ·從運行模式看，web頁面之間通過鏈接進行連接，從一個頁面跳到另一頁面構成頁面棧，瀏覽器中，用戶可能打開幾個這樣的頁面棧（即以tab標籤進行劃分）;android是一個多任務的操作系統，它的上面可以同時運行多個任務。每個任務都有一個界面棧，棧中的元素是界面組件對象的實例，其中負責與用戶交互的是前臺任務的棧頂組件。
